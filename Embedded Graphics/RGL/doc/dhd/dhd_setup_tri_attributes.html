dhd_setup_tri_attributes
Setup Utility Functions | Send Feedback
Summary

Setup triangle with attributes for rendering.

Syntax
dhd_bool_t dhd_setup_tri_attributes(dhd_handle_t a_device, const dhd_util_vertex_t * a_v0, const dhd_util_vertex_t * a_v1, const dhd_util_vertex_t * a_v2, dhd_prim_data_t * a_pout, dhd_edge_data_t * a_eout, dhd_uv_data_t * a_uvout, dhd_rhw_data_t * a_rhwout, dhd_depth_data_t * a_depthout, dhd_util_clipmode_t a_clipmode);
Description

This function can setup primitive information (bounding box), edges, UV, RHW and depth interpolants for any triangle. Depending on what pointers are passed the desired data is generated. The input vertices are passed as pointers to dhd_util_vertex_t, but depending on the output that is requested only a part of that structure has to contain valid information. If e.g. only primitive and edge edata is requested, only the m_x and m_y members need to contain data and thus pointers into an array of dhd_int32_t (casted to dhd_util_vertex_t) would work as well. 

 

The required members for each output are:

Requested output 
Required members of dhd_util_vertex_t 
pout (primitive out) 
m_x, m_y 
eout (edge out) 
m_x, m_y 
uvout (uv out) 
m_x, m_y, m_u, m_v 
rhwout (rhw out) 
m_x, m_y, m_rhw 
depthout (depth out) 
m_x, m_y, m_z 

 

All combinations of output data can be generated with the following restrictions : When edge data is requested primitive data has to be generated as well and clipping is applied only when primitive data is generated. It is possible (and actually usefull) to generate only UV/RHW data based on a triangle even when the mapped geometry is different (e.g. a quad or infinite plane). 

The recommended use is setting up a batch of triangles before rendering them with a single call to dhd_draw_array.

Parameters
Parameters 
Description 
a_device 
Driver context handle (as returned by dhd_init). 
a_v0 
First triangle vertex (see dhd_util_vertex_t) 
a_v1 
Second triangle vertex (see dhd_util_vertex_t) 
a_v2 
Third triangle vertex (see dhd_util_vertex_t) 
a_pout 
Pointer to dhd_prim_data_t receiving bounding box and edge control flags (can be null) 
a_eout 
Pointer to dhd_edge_data_t[3] receiving edge information (can be null) 
a_uvout 
Pointer to dhd_uv_data_t receiving uv information (can be null) 
a_rhwout 
Pointer to dhd_rhw_data_t receiving rhw information (can be null) 
a_depthout 
Pointer to dhd_depth_data_t receiving depth information (can be null) 
a_clipmode 
Specify to what region the geometry shall be clipped (see dhd_util_clipmode_t) 
Returns

DHD_TRUE if resulting triangle is visible DHD_FALSE if resulting triangle was entirely clipped and should not be drawn

Notes

Screenspace coordinates (m_x, m_y) are expected to be integers with 4bit subpixel precision while all vertex varyings are expected to be floats. This includes the depth (m_z). 

Setup works regardless of the triangle winding order (clockwise or counter clockwise) by automatically setting the edge inversion bits (E_DHD_INVERT) in case of a counter clockwise triangle.
